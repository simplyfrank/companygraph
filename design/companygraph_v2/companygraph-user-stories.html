P1 · Ontology Author

Maya — curates the schema

Owns the canonical model. Adds new node labels (e.g. Product, KPI), new edge types with allowed label pairs, and attribute schemas. Cares about consistency, defensibility, and zero-drift over time.

Surface
ontology-manager
Frequency
Weekly · ad-hoc
Success
Zero schema drift after 6 months; every type has a definition + example.
OA-1

Schema curation — define & evolve labels and edge types

ontology-manager

Extend the canonical schema beyond the six built-in labels and six edge types from graph-core/FR-03 and FR-04 without breaking existing data.

OA-1.1

As an Ontology Author, I want to add a new node label (e.g. Product) with a typed attribute schema, so that downstream tools can render and validate it.

  • A new label appears in /api/v1/stats keyset and is creatable via POST /api/v1/nodes/:label within one minute of registration.
  • Required attributes are enforced on POST + PATCH; violations return 400 {error:{code:"attribute_violation"}}.
  • Existing rows are not retroactively rewritten — historical attribute-less rows surface as null until backfilled.
OA-1.2

As an Ontology Author, I want to add a new edge type with explicit (fromLabel, toLabel) pairs, so that the validator rejects illegal pairs at write time.

  • Edge type definition stores allowed (fromLabel, toLabel) pairs; validator enforces them on POST /api/v1/edges. addresses C-02 (review pass 2)
  • Each new type is documented in ontology metadata and surfaces in /api/v1/schema.
  • Fixture proves rejection of a pair outside the allow-list with 400 {error:{code:"type_pair_violation"}}.
OA-1.3

As an Ontology Author, I want to deprecate a label or edge type without breaking existing data, so the graph stays internally consistent across schema revisions.

  • A deprecated type is marked deprecated_at but remains queryable.
  • Writes return 200 with a Deprecation response header; reads are unaffected.
  • Removal requires an explicit migration step that nulls or remaps all instances first; bare removal returns 409 {error:{code:"deprecation_required"}}.
OA-1.4

As an Ontology Author, I want to version the ontology, so I can roll back accidental changes.

  • Every schema change writes an immutable ontology_version row containing the diff, actor, and timestamp.
  • GET /api/v1/ontology/versions returns the version list with created_at, author, summary.
  • POST /api/v1/ontology/rollback/:version restores the prior schema in a single transaction; AC-04 idempotency rules preserved.
OA-2

Catalog management — keep the type catalog readable

ontology-manager

A single readable view of what exists, plus optional alignment to external retail reference models.

OA-2.1

As an Ontology Author, I want a single readable view of all node + edge types with descriptions and examples.

  • GET /api/v1/schema returns a stable JSON document covering all six base types plus any user-defined ones.
  • Per-type description and a one-line usage example are mandatory on creation; blank descriptions return 400.
OA-2.2

As an Ontology Author, I want to align my types with an external reference (e.g. ARTS retail), so the catalog stays defensible.

  • Each type may carry an external_alignment: [{source, id}] array.
  • Alignment is optional; absence does not block creation.
  • Querying /api/v1/schema?alignment=ARTS filters to aligned types only.
OA-2.3

As an Ontology Author, I want to import the catalog from a YAML/JSON file, so I can review schema changes in version control.

  • POST /api/v1/ontology/import is idempotent on type id and rejects schema-breaking changes.
  • GET /api/v1/ontology/export returns the same shape, round-trippable with zero diff.
OA-3

Schema safety — prevent destructive changes

ontology-manager

Dry-runs, refusal of data-invalidating changes, and an audit log of who changed what.

OA-3.1

As an Ontology Author, I want a dry-run mode for every schema change, so I see the impact before committing.

  • ?dryRun=true returns the planned changes without applying them.
  • Output includes affected node and edge counts per type.
OA-3.2

As an Ontology Author, I want to be warned when a change would invalidate existing data.

  • Adding a required attribute to a label with existing rows is blocked unless forceBackfill=true is supplied with a backfill value.
  • Block returns 409 {error:{code:"would_invalidate", details:{affected:n}}}.
OA-3.3

As an Ontology Author, I want every schema change to write a structured audit log entry.

  • One JSON-line audit row per change with actor, before, after, ts. aligns with FR-13 logging
  • Audit is queryable via GET /api/v1/ontology/audit?since=:iso.

PE-1

Browse & navigate the graph

process-explorer-ui

Drill from domains down to activities and pivot to bound roles, systems, and locations.

PE-1.1

As a Process Explorer, I want a domain index so I can see all four (or more) domains at a glance.

  • /explorer/domains lists every domain with member-journey count and last-updated time.
  • Clicking a domain card opens the journey list within 200 ms without a full reload.
PE-1.2

As a Process Explorer, I want to drill from a domain to its journeys, then a journey to its activities ordered by PRECEDES.

  • Journey detail renders activities in the order defined by PRECEDES, with arrows between consecutive steps.
  • Cycles in PRECEDES are flagged with a warning ribbon and rendered in declaration order; explorer never enters an infinite render loop.
  • Activity ordering survives reload (URL deep-link includes anchor).
PE-1.3

As a Process Explorer, I want to pivot from any activity to the roles, systems, and locations bound to it.

  • Activity detail renders four bound lists: roles (EXECUTES), systems (USES_SYSTEM), locations (AT_LOCATION), and adjacent activities (PRECEDES).
  • Each item routes to its entity detail; back navigation preserves scroll position.
PE-1.4

As a Process Explorer, I want a system-centric view: "show me all activities that use the POS system."

  • /explorer/systems/:id lists activities via USES_SYSTEM with their parent journey + domain.
  • A toggle reveals downstream INTEGRATES_WITH systems with one-hop neighbours visualised.
PE-2

Search & filter the graph

process-explorer-ui

Find anything by typing a fragment; combine filters; trace a path between two nodes.

PE-2.1

As a Process Explorer, I want full-text search across node names, so I can find anything by typing a fragment.

  • Search returns within 500 ms on the retail-mini seed (~60 nodes per FR-08).
  • Results are grouped by label.
  • Keyboard navigation: / focuses search; arrows move; enter opens.
PE-2.2

As a Process Explorer, I want to filter activities by system, role, or location simultaneously.

  • /explorer/activities?system=:id&role=:id&location=:id AND-filters correctly.
  • Empty filter returns all activities; URL is shareable + survives reload.
PE-2.3

As a Process Explorer, I want to find the shortest path between two nodes, so I see how they connect.

  • UI calls findPath(fromId, toId, maxDepth) (FR-07) and renders each hop with the edge-type label inline.
  • Depth selector caps at maxDepth = 8 NFR-09 with an inline error on overflow.
PE-3

Visualise & export

process-explorer-ui

Interactive canvas for one journey, plus PNG/SVG export and deep-linkable URLs.

PE-3.1

As a Process Explorer, I want an interactive graph canvas for a single journey, so I can present it in a meeting.

  • Canvas renders ≤ 200 nodes at 60 fps on a 2021 MacBook Air baseline.
  • Pinch-zoom on iPad Safari is captured by the canvas; the corresponding Native Conflicts row is added to process-explorer-ui requirements.
  • Selecting a node opens its detail in a side panel without losing canvas state.
PE-3.2

As a Process Explorer, I want to export the current view as PNG and SVG.

  • Export produces 1× and 2× PNG and a vector SVG; filenames embed the journey slug and ISO date.
PE-3.3

As a Process Explorer, I want deep-link URLs for every entity, so I can paste them into Slack.

  • #/explorer/journeys/:id, #/explorer/activities/:id, etc. all hydrate the correct detail panel from a cold load.
  • Invalid id renders a 404 panel with a back-to-domain link, not a blank screen.

Lin — asks in plain English

Domain head. Wants the conversational interface to be grounded in the graph — every answer cites nodes; never hallucinates retail facts; refuses unanswerable questions instead of confabulating.

chat-interface
Frequency
Daily · drive-by

CU-1

Ask questions in plain English

chat-interface

Natural-language Q&A grounded in the graph, with cited evidence and visible Cypher.

CU-1.1

As a Chat User, I want to ask "Which systems does Order Fulfillment use?" and get a grounded answer with cited nodes.

  • Response cites each system with its id and a clickable deep link to the explorer.
  • The generated Cypher query is shown on demand (collapsible by default).
CU-1.2

As a Chat User, I want the chat to refuse to answer questions outside the graph, so I never get hallucinated retail facts.

  • A question with no matching nodes returns "no nodes found in current graph" rather than a generated answer.
  • Out-of-scope questions ("what's the weather") return a fixed scope-redirect message naming the four downstream specs.
CU-1.3

As a Chat User, I want follow-up questions to retain context, so I can say "and which roles execute those?" after the previous answer.

  • The last answer's cited node ids are passed to the next prompt as bound context.
  • An explicit "new conversation" button clears the context.
  • Context bound count is displayed (e.g. "5 systems carried forward").
CU-1.4

As a Chat User, I want to see the source data in a side panel, so I can verify the chat's answer.

  • Citations open a side panel with the raw node + edge rows that supported the answer.
CU-2

Trust & guard-rails

chat-interface

Read-only Cypher, bounded results, observable latency.

CU-2.1

As a Chat User, I want every generated Cypher query to be read-only, so a bad question can never mutate the graph.

  • All chat-issued Cypher routes through POST /api/v1/query/cypher (read-only passthrough). NFR-06
  • Attempts to emit CREATE / MERGE / SET / DELETE are rejected upstream and surface as "this question is not answerable read-only — please use the explorer".
CU-2.2

As a Chat User, I want chat answers truncated to ≤1000 rows with a "see more in explorer" link.

  • Responses at the cap show the truncation banner with a count and explorer link. NFR-09
CU-2.3

As a Chat User, I want each answer to expose latency and Cypher execution time, so I can spot slow questions.

  • Response footer shows cypherDurationMs + totalMs, consistent with FR-13 logging.
CU-3

Conversation management

chat-interface

Bookmarks for repeated questions, shareable conversation URLs.

CU-3.1

As a Chat User, I want to bookmark useful questions, so I can re-run them weekly.

  • Bookmarked questions appear in a sidebar; click re-runs against the live graph.
  • Bookmark name is editable and free-text.
CU-3.2

As a Chat User, I want to share a conversation by URL, so a colleague can pick up where I left off.

  • Conversation deep-link restores message history and cited nodes.
  • Shared conversations are read-only for the recipient; they can fork to continue.

AN-1

System & integration overview

cto-analytics

A defensible map of which systems exist, how densely they integrate, and which serve which domains.

AN-1.1

As a CTO / Analyst, I want a system map showing every System node with its INTEGRATES_WITH neighbours, so I can see integration density.

  • Dashboard renders every system with degree-centrality and integration count.
  • Clusters are coloured by degree using shades of the single accent — no rainbow palette.
AN-1.2

As a CTO / Analyst, I want the domain ↔ system alignment matrix, so I can see which systems serve which domains.

  • Matrix rows are domains; columns are systems; cell value is the count of activities binding them via USES_SYSTEM.
  • Cells link through to the underlying activity list.
AN-1.3

As a CTO / Analyst, I want to flag duplicate-purpose systems (two systems used by the same activity), so I can prioritise consolidation.

  • A "consolidation candidates" panel lists activities with ≥ 2 USES_SYSTEM edges, sorted by usage count.
AN-2

Complexity & risk metrics

cto-analytics

Rank journeys by re-engineering value; expose critical paths.

AN-2.1

As a CTO / Analyst, I want a complexity score per journey (depth of PRECEDES chain × distinct systems × distinct roles), so I can rank journeys.

  • Score is computed nightly and rendered in a sortable table.
  • Each score's formula is shown on hover; score weights are tunable in settings.
AN-2.2

As a CTO / Analyst, I want a "single-system journey" report — journeys that go through one system end-to-end — for migration assessment.

  • Report enumerates journeys where distinct(USES_SYSTEM) == 1.
  • Each row links to the journey detail with the bound system pinned.
AN-2.3

As a CTO / Analyst, I want a critical-path report — the longest PRECEDES chain per journey.

  • Each journey row shows critical-path length, start activity, end activity.
  • Cyclic journeys are flagged but not crashed; longest acyclic chain reported.
AN-3

AI-leverage candidates

cto-analytics

Surface activities whose attributes mark them as automation candidates and produce an executive summary export.

AN-3.1

As a CTO / Analyst, I want to flag activities whose attributes mark them as repetitive=true and data_richness=high, so AI candidates are surfaced consistently.

  • Filter returns the candidate list; export to CSV is available.
  • Empty result handled with explicit "no attributes tagged yet — see ontology-manager" rather than a blank screen.
AN-3.2

As a CTO / Analyst, I want an executive summary export (PDF) covering top 5 journeys by complexity, top 3 consolidation candidates, top 3 AI candidates.

  • PDF is generated server-side from current graph state; reproducible across runs.
  • Header carries graph-state hash so a reader can verify the snapshot.

Bootstrap a domain catalog

process-explorer-ui (write)

Add journeys and activities through forms and bulk paste; bind to existing entities via typeahead.

As a Domain SME, I want to add a new UserJourney to my domain by filling a four-field form, so I don't need to write Cypher.

  • Form requires name, description, parent domain (pre-selected), and at least one activity stub.

As a Domain SME, I want to bulk-paste a list of activities under a journey, so I can transcribe a workshop output quickly.

SME-1.3

As a Domain SME, I want to bind activities to roles, systems, and locations from a typeahead, so I never type IDs.

  • "Create new" inline option creates the missing role/system/location and binds in one click.

Correct & validate

process-explorer-ui (write)

Mark doubtful entries for review; record who verified what; prevent out-of-domain edits.

SME-2.1

As a Domain SME, I want to flag a node as "needs review" with a reason, so doubtful entries surface in a backlog.

  • Node attribute review_status with values needs_review | verified | deprecated is settable via PATCH.
  • /sme/review-queue lists all needs_review nodes in my domain.
SME-2.2

As a Domain SME, I want to mark a journey as "verified by me on <date>", so accountability is visible.

  • Attributes verified_by (Role id) and verified_at (ISO date) are settable on a journey.

SME-2.3

As a Domain SME, I want a redirect when I land on an out-of-domain node, so I don't accidentally edit another team's catalog.

  • Read access is unaffected; SMEs can still browse other domains.

Quarterly sign-off

process-explorer-ui (write)

A single screen for the quarterly review with bulk sign-off.

SME-3.1

As a Domain SME, I want a quarterly checklist of every journey in my domain with sign-off state, so I can do the review in one sitting.

  • Checklist groups journeys by overdue (verified_at < 90 days ago) vs current.
  • Per-row toggle expands to show the activity list inline.
SME-3.2

As a Domain SME, I want to sign off journeys in bulk after spot-checking, so the workflow is realistic.

  • Multi-select plus "sign off selected" applies verified_by + verified_at in one transaction.
  • Audit log entry records the bulk action with the journey id list.

P6 · API Consumer

Theo — integrates from outside

External script developer. Pulls graph data into BI tools, syncs to a wiki, or seeds a separate audit log. Cares about stable contracts, typed clients, idempotent ingestion, and incremental sync.

Surface
REST · /api/v1/
Frequency
Daily · automated
Success
Integrations stable across schema revisions; idempotent imports.
API-1

Discover & navigate the API

graph-core

Published schema, consistent error envelope, versioned base path.

API-1.1

As an API Consumer, I want a published OpenAPI 3.1 schema at /api/v1/openapi.json, so I can codegen typed clients.

  • Schema covers every endpoint in FR-06, FR-07, FR-11 with request and response bodies.
  • Schema is generated from the same zod definitions used at runtime — no hand-maintained copy.
API-1.2

As an API Consumer, I want consistent error envelopes, so error handling is generic.

  • Every 4xx/5xx returns {error:{code, message, details?}}. NFR-05 · AC-20
  • Codes are stable across releases; each is documented in the OpenAPI schema.
API-1.3

As an API Consumer, I want a versioned base path, so my client survives schema evolution.

  • All endpoints under /api/v1/. FR-06
  • Breaking changes increment to /api/v2/ and /api/v1/ is supported in parallel for at least three months.
API-2

Read paths — query & export

graph-core

Full export, streaming export, and incremental sync.

API-2.1

As an API Consumer, I want bulk export of the full graph as JSON, so I can sync to a BI warehouse nightly.

  • GET /api/v1/export returns {nodes:[…], edges:[…]} deterministically ordered by id.
  • Payload is round-trippable through POST /api/v1/import with zero diff. NFR-04
API-2.2

As an API Consumer, I want streaming export via newline-delimited JSON, so I don't OOM on a 100k-node graph.

  • GET /api/v1/export.ndjson streams one node/edge per line with Content-Type: application/x-ndjson.
API-2.3

As an API Consumer, I want incremental sync — "give me everything changed since updatedAt > t" — so a downstream cache is cheap to keep current.

  • GET /api/v1/changes?since=:iso returns nodes + edges updated after the timestamp, ordered by updatedAt.
  • Tombstones for deleted ids are included in the response.
API-3

Write paths — ingestion

graph-core

Idempotent imports, partial-failure semantics, dry-run validation.

API-3.1

As an API Consumer, I want to push a delta payload via POST /api/v1/import and have it upsert idempotently on id.

  • Re-import of the same payload produces zero new nodes/edges. AC-08
  • Mixed-order payloads (edges before nodes) are handled via two-phase ingestion. FR-06
API-3.2

As an API Consumer, I want partial-failure semantics — one bad row should not abort the whole import — with a per-row error report.

  • Response shape is {accepted:n, rejected:[{index, error}]} rather than all-or-nothing.
  • Rejected rows do not roll back accepted rows.
API-3.3

As an API Consumer, I want a dry-run option on import, so I can validate before writing.

  • ?dryRun=true returns the same {accepted, rejected} shape without writing.

OPS-3

Team scorecards

A vs B vs C across cycle time, SLA adherence, complexity, cost. Same weights for every team.

OPS-3.1

As a VP Operations, I want a per-team scorecard aggregating activities executed by that team, so I can compare like-for-like.

  • Card shows mean cycle time, SLA adherence %, complexity score, runs/month — all rolled up from the team's bound activities.
  • Comparison view stacks teams side-by-side with consistent normalisation.

OPS-3.2

As a VP Operations, I want to export team scorecards to PDF for the monthly ops review pack.

  • Header carries graph-state hash + ontology version for audit traceability.

P8 · CFO / FinOps Partner

Diego — translates graph to money

Reads every node as a cost. Wants $ per journey run, vendor concentration risk, and a defensible automation ROI built from FTE-minutes and loaded rates — not an abstract "leverage score".

Surface
exec-analytics (Finance)
Frequency
Monthly · quarterly board pack
Success

FIN-1

Cost-to-serve per journey

exec-analytics

Roll up FTE-minutes × loaded_role_rate + amortised system cost into $/run and $/year.

FIN-1.1

As a CFO, I want every journey detail to show cost_per_run_sgd, runs_per_month, annual_cost_sgd, derived from bound activities + roles + systems.

FIN-1.2

As a CFO, I want a cost-band heatmap across all journeys, so the most expensive surfaces are obvious without sorting.

  • Heatmap coloured in shades of the single accent — no rainbow palette.

FIN-2

Vendor concentration & renewal radar

exec-analytics
