  (`Blocked by` / `Blocks`); no out-of-order execution.

  helper (T-04) run through the loopback API on `127.0.0.1:8787`, so their
  **integration** tests need a running API + Neo4j (+ Postgres for T-04/AC-19).

  storage-level assertion needing no route surface; the full integration files
  run green under `bun test:integration` once the stack is up (`bun run dev`).

### T-07 — Shared System + Role catalog ensure (direct-driver MERGE on `operatorSeedKey`)

- **Files** (1): `api/src/seed/ensure-catalog.ts` (new — `ensureSystems` + `ensureRoles`)
- **Implements**: design §3.3, §4.3 — closes AC-04 (+ AC-05 role half); supports FR-04, FR-05, NFR-01, NFR-02
- **Complexity**: moderate
- **Blocked by**: T-01
- **Blocks**: T-08, T-09
- **Steps**:
  - `ensureSystems(driver)` — for each `SYSTEMS` row (T-01), MERGE a `System` on
    the **top-level `operatorSeedKey`** marker (B-01, §4.3), never bare `name`:
    ```cypher
    MERGE (s:System {operatorSeedKey: $seedKey})
    ON CREATE SET s.id=$id, s.name=$name, s.description=$desc,
                  s.createdAt=$now, s.updatedAt=$now, s.attributes_json=$attrs
    ON MATCH  SET s.name=$name, s.description=$desc, s.updatedAt=$now,
                  s.attributes_json=$attrs
    RETURN s
    ```
    `attributes_json` carries `{systemKind, seedKey}` (systemKind set explicitly
    so the System registry attribute check passes). Because the MERGE key is
    `operatorSeedKey` (which the retail `System {name:"CRM"}`,
    `retail-mini.json:69`, lacks), the operator `CRM` is a **new, separate** node
    and the retail `CRM` is never read/matched/written. **C-07: Systems
    converge attributes on re-seed** (`ON MATCH SET attributes_json`).
  - `ensureRoles(driver)` — for each `ROLES` row, create the operator core
    **`:Role`** process node (NOT `:RBACRole`) via the established seed-script
    direct-driver MERGE pattern (`seed-rbac-roles.ts:157` shape, a trusted
    operator-tooling write that bypasses the router gate by design, Rule A):
    ```cypher
    MERGE (r:Role {operatorSeedKey: $seedKey})
    ON CREATE SET r.id=$id, r.name=$name, r.description=$desc,
                  r.createdAt=$now, r.updatedAt=$now, r.attributes_json=$attrs
    ON MATCH  SET r.name=$name, r.description=$desc, r.updatedAt=$now
    RETURN r
    ```
    `attributes_json` carries `{seedKey}`. **C-07: Roles keep first-written
    attributes on re-seed** — the `ON MATCH SET` deliberately omits
    `attributes_json`. Seed **no `:RBACRole`**; add no permission string
    (FR-12). `operatorSeedKey` is a plain property — no `NODE_LABELS`/`EDGE_TYPES`
    edit, no new constraint (NFR-01).
- **Verification**: `api/__tests__/saas-operator-catalog.integration.test.ts`
  (systems+roles portion) — seven Systems seeded once, each with a stable id +
  valid `systemKind`, shared (not per-domain); a re-run yields no duplicate
  (`operatorSeedKey` MERGE); the operator `CRM` is a distinct node carrying
  `operatorSeedKey:"crm"` + `systemKind`; **the retail `CRM` node is untouched**
  — its id + description unchanged and it carries **no** `operatorSeedKey` after
  `seed:saas-operator` (B-01, AC-04). Roles seeded as **`:Role`** (not
  `:RBACRole`), re-run no duplicate; a re-seed leaves a Role's `attributes_json`
  as first-written (C-07). Deferred-green: `bun test:integration`.

### T-09 — Directory-iterating seed loader (CLI entrypoint)

- **Files** (1): `api/scripts/seed-saas-operator.ts` (new)
- **Implements**: design §4.4, §7 — closes AC-06, AC-07, AC-08; supports FR-07, FR-08, FR-09, NFR-02
- **Complexity**: complex
- **Blocked by**: T-02, T-05, T-06, T-07, T-08
- **Blocks**: —
- **Steps**: The `bun run seed:saas-operator` CLI entrypoint (§4.4):
  1. **Step (a) — ensure scaffold** (always, regardless of directory contents):
     `ensureOperatorRoot` → `ensureFunctionDomains(root.id)` → `ensureSystems` →
     `ensureRoles` → `ensurePersonas(base)`. This is why an empty directory is a
     clean no-op, not an error (the scaffold is still ensured, AC-06/Risk 5).
  2. **Step (b) — discover + load slices**:
     ```ts
     const dir = resolve(import.meta.dir, "../../shared/seed/saas-operator");
     const files = existsSync(dir)
       ? readdirSync(dir).filter((f) => f.endsWith(".json")).sort()
       : [];
     for (const f of files) {
       const body = readFileSync(resolve(dir, f), "utf8");
       const res = await fetch(`${base}/api/v1/import`, {   // POST /api/v1/import (B-02)
         method: "POST",
         headers: { "content-type": "application/json" },
         body,
       });
       // 409 model_lifecycle_route_required → malformed fixture; surface + fail (AC-08)
     }
     ```
     Slices post to **`POST /api/v1/import`** (`realImport`, the `{nodes,edges}`
     process-content route that carries the lifecycle guard) — **never**
     `POST /api/v1/ontology/import` (B-02). `filter(".json")` skips `.gitkeep`,
     so a `.gitkeep`-only directory yields zero `*.json` and step (b) is a clean
     no-op. Load order is sorted filename order; step (a) always precedes step
     (b), so a slice referencing a shared catalog id always runs after the
     catalog is ensured (FR-08). **Adding `marketing.json` requires no edit to
     this file** — it is discovered (AC-06). The loader never deletes and touches
     only the operator subgraph + shared catalog (FR-08/NFR-02). Surface a
     non-2xx import (including the 409 guard) as a script failure.

- **Implements**: design §6.4 + `review-design.md` C-06/N-04/N-05 — closes AC-10, AC-11, AC-12, AC-13, AC-14 (tsx), AC-15; supports FR-14, FR-15, NFR-06

    **never** re-implemented) and resolve the SaaS-Operator root as subject,
    defaulting to it via the OQ-1 `name:"SaaS Operator"` + `saasOperatorRoot:true`
    marker key if the active model is something else (FR-15).
  - **Read** — one `POST /api/v1/query/cypher` (`runPassthrough`, read-only,
    `query:read`, C-01) fetching the six `IN_MODEL` domains + a per-domain
    descendant count **filtered to journeys/activities** (C-03):
    ```cypher
    MATCH (m:BusinessModel {id:$operatorRootId})
    MATCH (d:Domain)-[:IN_MODEL]->(m)
    OPTIONAL MATCH (d)<-[:PART_OF*1..]-(desc)
    WHERE desc:UserJourney OR desc:Activity
    WITH d, count(DISTINCT desc) AS journeyActivityCount
    RETURN d.id AS id, d.name AS name, d.description AS description,
           d.attributes_json AS attributes_json, journeyActivityCount
    ORDER BY d.name
    ```
    Any `runPassthrough` failure — timeout, cap hit, or non-2xx envelope — maps
    to the **error** state (C-04), never a silent empty.
  - **States (UX-01, catalog-first N-04/N-05)** — use the exported catalog
    primitives from `pwa/src/views/_shared.tsx`:

    - **empty** (AC-12) → `EmptyState` when the model resolves but returns zero
      domains: a prompt to run `bun run seed:saas-operator`.
    - **error** (AC-13) → `ErrorState` + a `Button` retry that refetches.
    - **ready** (AC-10) → a keyboard-reachable grid of six function cards (name +
      description + `journeyActivityCount`), each a native link that deep-links
      into Explorer for that domain (`#/explorer/domain-detail/<domainId>` via
      `toHash`).
  - **Accessibility (AC-15, N-04)** — the view root is the catalog **`ViewRegion`**
    landmark (not a raw `<section>`); the six function links are native
