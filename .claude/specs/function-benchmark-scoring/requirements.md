# Requirements: function-benchmark-scoring

## Summary

`function-benchmark-scoring` is a **wave-3** feature of the SaaS-Operator
business-process model (blueprint `.claude/specs/blueprint-saas-operator.md`,
milestone M3). For the **SaaS-Operator `BusinessModel` root** it computes a
**per-function descriptive maturity score** across the six function domains
(Marketing, Sales, Finance & Accounting, Customer Success, Product & Delivery,
Platform Ops) and surfaces it in the **`BenchmarkReport`** view at
`#/insights/benchmarks` (blueprint View Tree, verbatim; route pre-registered by
`saas-operator-foundation` per XD-05). Each function's maturity is a composite
of three descriptive sub-scores:

1. **Metric-vs-benchmark** â€” for the function's KPIs that ground in a canonical
   `saas-metric-library` `MetricDefinition` (via the `MEASURES` edge,
   XD-06/XD-06-erratum), how the KPI's latest measured value compares to the
   metric's `benchmark` descriptor.
2. **Key-activity coverage** â€” how completely the function's activities are
   modeled and measured (share with an executing role, a used system, and KPI
   coverage). Where a human has already marked activities *key* via
   `key-activity-optimizer`, an **optional** marked-key-covered component is
   added; that mark is usually empty for a freshly-seeded operator model, so it
   is applicability-flagged, not a scored miss (B-03).
3. **System-augmentation level** â€” how augmented the function's activities are,
   read from `USES_SYSTEM` coverage weighted by the used systems' `systemKind`
   from the **closed 3-value vocabulary** `{functional, agentic, ai_predictive}`
   (`shared/src/schema/system-kind.ts`, `system-augmentation-model`). This is an
   augmentation/AI-leverage axis (`functional â†’ agentic â†’ ai_predictive`), **not**
   a manual-vs-automated axis â€” a `functional` system is a real software system,
   not a "manual" activity (B-01).

Per blueprint XD-11 (and the feature's own scope: *"descriptive only, like
studio XD-11"*) the report is **descriptive** â€” scores, sub-scores, and the
evidence behind them â€” and **carries no prescriptive recommendation** ("improve
X", "automate Y") anywhere in its wire shape or view. It is **read-only**: it
computes on request over the existing graph and writes nothing.

It **reuses the `key-activity-optimizer` scoring *pattern*** (not its mark data,
which is usually empty here â€” B-03): a pure, I/O-free scoring module fed by a
model-scoped Neo4j read, a `meta`-carrying deterministic response, and a
read-only PWA view with all four states â€” but scopes and aggregates **per
function domain** rather than per activity.

Ownership boundaries and out-of-scope surfaces are enumerated once in **Scope
Boundaries** (this feature owns only its `BenchmarkReport.tsx` view file + CSS
module and the single `benchmarks:` line in `views/index.tsx`'s `VIEWS` map; it
authors no function content and edits no KPI/metric/risk/SLA/route-registration
code owned elsewhere â€” N-01).

## Motivation

1. The blueprint's M3 goal is *"the operator can see cross-function health +
   function maturity"* (Build Order). `cross-function-exec-rollup` delivers the
   **health cockpit**; this feature delivers the **function-maturity** half â€” a
   descriptive read on *how well each function is modeled and measured against
   canonical SaaS benchmarks*, which the cockpit does not compute.
2. Once the six content specs (wave 2) have authored each function's journeys,
   activities, roles, systems, and metric-grounded KPIs, there is enough
   structure to ask a comparative question: *is this function's CAC / NRR /
   MTTR at or off its canonical benchmark, are its activities modeled and
   measured, and how automated is it?* Answering that per function, side by
   side, is the operator's maturity read.
3. The `saas-metric-library` gives every operator KPI a **canonical benchmark**
   (the frozen roster's `benchmark` descriptor per metric â€” XD-06). Without a
   shared benchmark, "how does this function compare" is unanswerable; with it,
   a descriptive metric-vs-benchmark verdict per KPI, aggregated per function,
   becomes possible. This feature is the first consumer of the `benchmark`
   field for comparison.
4. `key-activity-optimizer` already established the **descriptive-scoring**
   pattern for the studio (pure `derive/` scorer + model-scoped read + read-only
   board, XD-11 â€” no prescriptive output). This feature reuses that exact
   contract, aggregated per function, so the two descriptive-scoring surfaces
   stay consistent and auditable.
5. The blueprint View Tree assigns `#/insights/benchmarks` â†’ `BenchmarkReport`
   to this spec; `saas-operator-foundation` already registered that route
   (rendering `BusinessTabPlaceholder`). This spec replaces the placeholder with
   the real report.
