# /wireframe-extract — HTML mock → manifest → component catalog

Turn the project's `./design/companygraph/` folder (static HTML mocks +
distilled `DESIGN.md`) into a managed, idempotent component catalog
consumed downstream by `/stitch` (generation) and `/design-apply`
(application). Re-running reconciles a manifest instead of clobbering
work, and reports drift between the mocks and what was extracted.

Ported from `hautemaison/.claude/skills/design-to-storybook` with two
adaptations for companygraph:
1. Target framework is **Vite + React + TypeScript** (companygraph's
   stack), not `@storybook/html`. Output components land in
   `pwa/src/views/` and `pwa/src/components/`, with stories optional.
2. The design-system reference is the **single `DESIGN.md`** at
   `design/companygraph/DESIGN.md` (not a multi-file
   `design/<DS>/src/{tokens.css, design-system.yaml, …}` layout).

## Usage

- `/wireframe-extract` — full pipeline: discover → configure → extract → scaffold → verify.
- `/wireframe-extract --discover` — phases 1–2 only. Write/refresh the manifest, no code generated.
- `/wireframe-extract --component <name>` — re-extract a single catalog entry.
- `/wireframe-extract --reconcile` — re-run discovery against the existing manifest, report drift, regenerate only stale entries. Run this after the SessionStart drift hook fires.
- `/wireframe-extract --verify` — assert the generated components import cleanly and pass typecheck; no extraction.

## Source-of-truth layout (the input contract)

Companygraph's wireframe set:

- `design/companygraph/*.html` — full-page **mocks** (the screens). Components are inlined; the catalog is mined from here.
- `design/companygraph/DESIGN.md` — distilled design tokens, type ramp, layout primitives, component vocabulary, and 27-screen catalogue. **Names, variant vocabulary, and token names come from here**.
- `design/companygraph/*.png` — supporting screenshots (optional reference; not parsed).

Token CSS for the React app is generated by
`scripts/stitch-tokens-to-css.ts` from `.claude/stitch/design-system.yaml`
(which `/wireframe-extract --discover` produces from the mocks' inline
`<style>` blocks on first run). The OKLCH custom properties declared in
the mock `<style>` blocks are the **only** styling source — components
must reference these tokens, never the resolved colour/size values.

If the design folder layout deviates from the above, infer the
equivalents and record the resolved paths in `manifest.paths` —
downstream phases read paths only from the manifest.

## The manifest (what this skill manages)

Two files at repo root, both owned by this skill, both committed:

- **`design-system.manifest.yaml`** — human-readable config + component catalog. Reconciled, never blindly overwritten.
- **`design-system.manifest.lock`** — `shasum -a 256` baseline of every tracked design input, in `shasum -c` format. The drift hook's baseline. Regenerated wholesale each run.

Manifest schema:

```yaml
meta:
  generatedAt: <ISO8601>
  builder: "vite + @vitejs/plugin-react"
  designSystemVersion: "<x.y.z>"
paths:
  mocks: ["design/companygraph/*.html"]
  designMd: "design/companygraph/DESIGN.md"
  tokensYaml: ".claude/stitch/design-system.yaml"
  tokensCss: "pwa/src/styles/tokens.css"
  output:
    views: "pwa/src/views"
    components: "pwa/src/components"
    stories: "pwa/src/components"   # *.stories.tsx colocated, optional
components:
  - name: TopBar                          # from DESIGN.md §5, PascalCase
    sources: ["design/companygraph/companygraph-views.html"]
    region: ".topbar"                     # selector the markup was lifted from
    variants: ["default", "compact"]      # per media query in mock CSS
    props: [{ name: "env", type: "string" }, { name: "user", type: "{name:string,initials:string}" }]
    status: extracted                     # pending | extracted | reviewed
    target: "pwa/src/components/TopBar.tsx"
    notes: ""                             # HUMAN-OWNED. never overwritten by the skill.
    overrides: {}                         # HUMAN-OWNED. wins over auto-derived fields.
coverage:
  mocks: 4
  components: 0
  views: 0
  pending: []
```

**Idempotency rules — non-negotiable** (carry-over from hautemaison contract):

- `notes` and `overrides` on an existing component entry are preserved verbatim across runs. Auto-derived fields (`sources`, `region`, `variants`, `props`) are refreshed *unless* shadowed by a key in `overrides`.
- `status: reviewed` means a human signed off — never downgrade it; if its sources changed, keep `reviewed` but append a `⚠ source changed` marker to `coverage.pending` (not into `notes` itself) and regenerate the component file.
- A component removed from the mocks is **not** deleted. Set `status: orphaned`, leave the files, list it under `coverage.pending`, and report it. Deletion is a human decision.
- The `.lock` is rewritten every run from the current design inputs so the drift hook's next baseline is accurate.

## Execution protocol

### Phase 1 — Discover

1. Resolve the input contract paths; write them to `manifest.paths`. If `./design/companygraph` is absent, stop and say so.
2. Read the naming authority: `DESIGN.md` (sections 5 "Component vocabulary" and 6 "Screen catalogue"). Build the canonical component name/variant list. Names come from here.
3. Inventory the screens listed in `DESIGN.md` §6 — each `data-view`/`data-tab` is a near-certain page boundary with its caption already authored.
4. Scan the inline `<style>` blocks of the mock HTML files for `:root` custom-property declarations. These are the design tokens. If `.claude/stitch/design-system.yaml` does not yet exist, generate it now from those declarations + the `DESIGN.md` tables.
5. Scan mock `<body>` markup for recurring class clusters that map to vocabulary names. Cross-reference, don't guess: a structure with no vocabulary name is reported as an "unnamed candidate," not auto-named.

### Phase 2 — Configure

1. Load existing `design-system.manifest.yaml` if present. Diff the discovered catalog against it.
2. Reconcile per the idempotency rules above. Produce a change summary: `added / refreshed / unchanged / orphaned / pending-review`.
3. Write `design-system.manifest.yaml`. Regenerate `design-system.manifest.lock` from all tracked inputs (`shasum -a 256` over every file under `design/companygraph/`).
4. If invoked with `--discover`, stop here and print the change summary + the catalog table.

### Phase 3 — Extract

For each component with `status != reviewed` (or the one named by `--component`):

1. Lift its markup from the highest-confidence source listed in the manifest. Strip page chrome; keep only the component subtree.
2. Translate to React 18 + TypeScript JSX. Replace inlined literal style values with the OKLCH tokens declared in the mock's `<style>` block — never hardcode a colour or size that has a token name.
3. Parameterize text/image/href into the `props` from the manifest.
4. Emit `<output.components>/<Name>.tsx` exporting a single React FC. Use CSS Modules (`<Name>.module.css`) for component-scoped rules. Token references go through `var(--…)`.
5. (Optional) Emit colocated `<Name>.stories.tsx` if Storybook is configured. Otherwise this is skipped.
6. Set `status: extracted`. Update `coverage`.

Deduplicate: if the same structure appears across mocks, extract once; list every source in `sources`.

### Phase 4 — Scaffold target

Companygraph already has `pwa/` scaffolded — this phase is normally a no-op. If `pwa/src/styles/tokens.css` is missing, run `bun run scripts/stitch-tokens-to-css.ts` to generate it from `.claude/stitch/design-system.yaml`.

### Phase 5 — Verify

1. `bun run typecheck` — must pass.
2. Run `bun run scripts/design-conformance.ts --view <newly-extracted-file>` for each component touched this run. Hard-fail on any `FAIL` finding.
3. Report a coverage table: mocks discovered → components in catalog → React files emitted, plus `pending` (orphaned / unnamed-candidate / source-changed) as an explicit action list.

## Guardrails

- The skill **reads** `./design/companygraph`; it never edits the mocks or DESIGN.md. Output goes only to `pwa/src/`, `.claude/stitch/design-system.yaml`, and the two manifest files.
- Tokens are sacred: components consume CSS custom properties declared in tokens.css. A PR that hardcodes a hex/oklch value lifted from a mock is a defect — flag it in Phase 3.
- Vocabulary is sacred: a component's name/variants must trace to `DESIGN.md` §5/§6. Anything unnameable is a reported candidate, not an invention.
- Never delete a human's `notes`, `overrides`, or a `reviewed` status. When in doubt, mark `pending` and report — let the human decide.
- See `.claude/patterns/design-system-extract.md` for the heuristics and the manifest/lock/drift-hook contract.

## Relationship to the rest of the pipeline

```
design/companygraph/*.html + DESIGN.md
        │
        ▼
 /wireframe-extract  (this skill)
        │
        ├─▶ design-system.manifest.yaml   (catalog + props)
        ├─▶ design-system.manifest.lock   (drift baseline)
        ├─▶ .claude/stitch/design-system.yaml   (tokens — input to /stitch)
        └─▶ pwa/src/{views,components}/*.tsx    (initial extraction)
                │
                ▼
        scripts/stitch-tokens-to-css.ts  ──▶  pwa/src/styles/tokens.css
                │
                ▼
        /stitch generate-page <data-view>/<data-tab>   (PA skill, already inherited)
                │
                ▼
        /design-apply  (PA skill, already inherited)
                │
                ▼
        scripts/design-conformance.ts   (hard gate)
                │
                ▼
        merge into pwa/src/views/*.tsx
```
